# LogseqBrain v0.6.0 — Design

**Status:** Draft, pending implementation
**Date:** 2026-05-01
**Target version:** 0.6.0 (from 0.5.0)
**Release shape:** Single bundled minor version (Approach 1 from brainstorm)

## Goals

1. **Journey log** — one-line activity trail in today's journal for every brain skill use, alongside the existing rich save-time summary.
2. **Token frugality** — measurable reduction in the always-loaded skill body footprint via progressive disclosure, plus leaner load and save paths.
3. **Revision pass** — tighten skill description frontmatter for better triggering precision; refresh `ROADMAP.md` to match shipped reality and inform future planning with the Logseq roadmap.

## Out of scope

- Storage abstraction layer to support Logseq's DB version. The DB version is in beta; "Markdown Mirror" is planned, so markdown isn't going away. Defer until Logseq's direction stabilizes.
- Logseq DB plugin API integration. API is in development per Logseq's roadmap — too early to build on.
- Headless sync via Logseq's new CLI. Not GA. Revisit later.
- README updates for v0.6.0 features. Apply at release time, not part of design.
- Automated tests. The project remains validation-by-hand against a scratch graph (per CLAUDE.md).

## Architecture overview

Four skills (`brain-init`, `brain-load`, `brain-save`, `brain-status`) currently each carry ~70-180 lines of inline flow logic with significant duplication (path resolution, staleness rules, soon-to-be journey logging). v0.6.0 splits each `SKILL.md` into a compact orchestrator + `references/*.md` files loaded on demand, and introduces a `skills/_shared/` directory (sibling to the skill folders) for logic used by multiple skills.

New top-level layout:

```
skills/
├── brain-init/
│   ├── SKILL.md                          # ~50-80 lines: orchestration only
│   └── references/
│       └── templates.md                  # consolidated page templates
├── brain-load/
│   ├── SKILL.md
│   └── references/
│       ├── matching.md                   # fuzzy match algorithm
│       └── search.md                     # cross-graph search
├── brain-save/
│   ├── SKILL.md
│   └── references/
│       ├── categories.md                 # six save categories
│       ├── auto-suggest.md
│       └── decisions.md                  # conflict + cross-project detection
├── brain-status/
│   └── SKILL.md
└── _shared/                              # new: shared references
    ├── path-resolution.md                # host-aware graph path resolution
    ├── journey-log.md                    # journey-trail write logic
    └── staleness.md                      # stale-project rules (used by load + status)
```

Note: `_shared/` lives at the `skills/` level, sibling to each skill folder, not inside any skill's `references/`. Each `SKILL.md` reads shared content via the relative path `../_shared/<file>.md` (or absolute path resolved from the plugin root). This avoids cross-platform symlink concerns on Windows.

## Section 1 — Journey log

### Mechanism

Each brain skill, after completing its main work, appends one bullet to a `## Activity` section in today's journal `journals/yyyy_MM_dd.md` (date with underscores).

The existing `## Sessions` section (rich save-time summary) is **kept unchanged**. The two sections coexist with distinct purposes:

- `## Sessions` — what work was done (project-linked, written by `brain-save` only)
- `## Activity` — when the brain was touched and how (terse one-liner per event, written by every skill)

### Format

> **⚠ Superseded by v0.7.0** — first-dogfood evidence (2026-05-02) showed Logseq does NOT visibly auto-render `created-at::` as inline timestamps, contrary to the rationale below. Without prefixes the activity trail loses its quick-scan chronological signal. v0.7.0 will re-add an `HH:mm` prefix to activity-line writes. The rest of this section (file structure, surgical-Edit pattern, events table) stands. See `docs/superpowers/specs/2026-05-31-v0.7.0-design.md` for the reversal item.

Bullet per event. No time prefix — Logseq auto-assigns `created-at::` block metadata when the file is parsed.

```markdown
- ## Activity
  - loaded [[Projects/LogseqBrain]] (brief)
  - saved [[Projects/LogseqBrain]]
  - searched "strategy pattern" · 3 hits
  - viewed dashboard
  - loaded [[Projects/ChivalricQuest]] (full)
```

### Events covered

| Skill        | Trigger event       | Activity line                              |
|--------------|---------------------|--------------------------------------------|
| `brain-init` | graph init          | `initialized graph at <path>`              |
| `brain-init` | project add         | `created project [[Projects/X]]`           |
| `brain-load` | project load        | `loaded [[Projects/X]] (brief\|full)`      |
| `brain-load` | overview            | `loaded brain overview`                    |
| `brain-load` | search              | `searched "term" · N hits`                 |
| `brain-save` | session save        | `saved [[Projects/X]]`                     |
| `brain-status` | dashboard         | `viewed dashboard`                         |

### File-creation behavior

- If today's journal does not exist: create it with both `## Sessions` and `## Activity` sections present (empty), then append.
- If `## Activity` section is missing from an existing journal: insert it directly after `## Sessions` (or at end of file if neither exists).
- All writes use the Edit tool with surgical anchoring — never rewrite the whole journal.

### Config toggle

`.brain-config.json` schema extends to:

```json
{
  "graphPath": "/path/to/ClaudeBrain",
  "journeyLog": true
}
```

Default `journeyLog: true` (enabled). Skills read config once during path resolution and skip the activity write if explicitly set to `false`. The toggle exists as an escape hatch; not exposed as an env var (YAGNI for v0.6.0).

### Side-effect note

Adding journey logging makes `brain-load` and `brain-status` produce write side-effects on otherwise read-only operations. This is acceptable because:

1. The user invoked the skill explicitly — this is not auto-persistence.
2. The write is tiny (one bullet line) and atomic.
3. The toggle exists for users who object.

This does NOT relax the project's "auto-save is a suggestion only" invariant — `brain-save` itself still requires explicit user trigger.

### Implementation contract

The journey-log write is encapsulated in `skills/_shared/journey-log.md`. Every skill calls into it with a single string argument (the activity line). The reference doc specifies:

- Input: activity-line string (without leading `- ` bullet marker)
- Behavior: ensure today's journal exists with `## Activity`, append bullet, surgical Edit
- Side effects: file write to `journals/yyyy_MM_dd.md`
- Failure mode: if config has `journeyLog: false`, no-op silently

## Section 2 — Token frugality

Three layers, in priority order.

### 2.1 Progressive disclosure

Move detailed flows from `SKILL.md` into per-skill `references/*.md` and the cross-skill `skills/_shared/*.md`. Target each `SKILL.md` at ~50-80 lines (orchestration + decision tree only).

| File                                | Currently | Target  |
|-------------------------------------|-----------|---------|
| `brain-init/SKILL.md`               | 104       | 60-70   |
| `brain-load/SKILL.md`               | 124       | 70-80   |
| `brain-save/SKILL.md`               | 178       | 80-90   |
| `brain-status/SKILL.md`             | 67        | 50-60   |

`SKILL.md` content split:

| Skill        | Stays in SKILL.md                                                | Moves to references                                                                                          |
|--------------|------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------|
| brain-init   | first-time init flow, project-add flow, pointer to path resolution | `references/templates.md` (Index, Meta, Decisions, Project templates consolidated)                          |
| brain-load   | mode selection (brief/full), top-level flow                      | `references/matching.md` (fuzzy match), `references/search.md` (cross-graph search)                          |
| brain-save   | top-level save flow, journal write, journey-log write call       | `references/categories.md` (six save categories), `references/auto-suggest.md`, `references/decisions.md`    |
| brain-status | top-level dashboard flow                                         | (none skill-specific; uses `_shared/staleness.md`)                                                          |

Shared references (used by 2+ skills, deduped under `skills/_shared/`):

- `skills/_shared/path-resolution.md` — host-aware path resolution (Cowork vs. Claude Code/Copilot/Gemini)
- `skills/_shared/journey-log.md` — journey-trail write logic
- `skills/_shared/staleness.md` — stale-project rules (used by `brain-load` and `brain-status`)

### 2.2 Leaner load paths

`brain-load` currently reads the full project page even in brief mode. Two changes:

- **Section-targeted reads**: Grep for `## <SectionName>` to locate offset, then `Read` with `offset+limit` to grab just that section. Especially valuable for projects with long Session Logs.
- **Defer cascading reads in brief mode**: don't read linked `[[Page]]` references or linked Jira plans unless full mode or the user explicitly asks. Brief mode = project-page sections only.

### 2.3 Smarter save reads

`brain-save` currently reads the entire project page to find append points. Replace with:

- **Grep + targeted Read**: locate each section's offset and trailing boundary (next `##` heading); read just enough to anchor the Edit, not the whole file.
- For projects with long Session Logs, this is the difference between reading 1500 lines vs. 80.

Constraint: Edit's `old_string` matching must remain unique within its anchored window. For multi-paragraph appends to a section, may need to read the section + a few following lines.

### Token-savings target

Combined target: ~50% reduction in always-loaded skill-body footprint, plus measurable load/save token reduction on real projects. Validated by manual comparison (load same project before/after).

## Section 3 — Revision pass

### 3.1 Description tuning

Current frontmatter descriptions average ~70-90 words each (verbose trigger lists). Tighten each to ~30-50 words: lead with purpose, terse trigger set, explicit scope boundary.

Sample shape (apply consistently to all four):

```yaml
description: >
  Load project context from the Claude Brain Logseq graph. Triggers: "load brain",
  "load <project>", "resume <project>", "what do we know about <topic>", "continue
  work on <project>". Don't fire for write operations (use brain-save) or generic
  questions about Logseq itself.
```

Coverage gaps to close:

- `brain-load` — add: "resume X", "open X", "switch to X"
- `brain-save` — add: "log this", "before I quit", "wrap up"
- `brain-status` — add: "show brain", "summary"

Net effect: ~40-50% fewer tokens in always-loaded skill index, plus better triggering precision.

## Validation plan

No automated tests in this repo. Validation = manual round-trip against a scratch graph (per CLAUDE.md).

Checklist:

1. Init scratch graph (`brain-init` against empty folder).
2. Add a project (`brain-init project ScratchProject`).
3. Save a session (`brain-save` with some made-up decisions and plan items).
4. Load brief (`brain-load ScratchProject`) — verify section-targeted read.
5. Load full (`brain-load ScratchProject full`) — verify cascading reads work in full mode.
6. Status (`brain-status`) — verify dashboard flow.
7. Search (`what do we know about X`) — verify cross-graph search.
8. Verify journey log: each of steps 1-7 added one bullet under `## Activity` in today's journal.
9. Set `.brain-config.json` `{"journeyLog": false}`, repeat any step, verify no `## Activity` write.
10. Compare token usage on `brain-load` (brief) before/after — directionally lower.

Document this checklist in `CONTRIBUTING.md` as the v0.6.0 manual validation procedure.

Captured in `ROADMAP.md` per Section 3.2. Summary:

- **Storage abstraction layer**: defer until Logseq's DB version stabilizes. "Markdown Mirror" being planned means markdown stays viable.
- **DB plugin API integration**: wait for API GA. Then evaluate replacing file IO with API calls — preserves the same skill surface, swaps the backend.
